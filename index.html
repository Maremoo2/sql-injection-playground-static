SQL Injection Playground — simulated (safe)

This is a client-side simulation. No real database or server is used. Safe to publish publicly.

Try these: Type a correct password (e.g. wonderland) to log in, or experiment with an always-true expression (e.g. 2+2 = 4 or ' OR '1'='1) to see the vulnerable simulation bypass. A wrong password will be denied by the safe check.

Tip: Open DevTools → Console to see the constructed SQL (useful for technical viewers).

Vulnerable — constructed SQL (simulated)


          

          
This panel shows the SQL that would be constructed by unsafe string concatenation. If the WHERE clause contains an always-true expression (e.g. 2+2=4), the simulation treats it as a bypass.

  1. Knappen «Sårbar» viser hvordan en applikasjon som setter sammen brukerinput direkte kan bygge sårbar SQL. Demoen simulerer hva som skjer hvis databasen evaluerer WHERE-betingelsen — den oppdager alltid-sanne uttrykk som matematiske likninger (2+2=4) og typiske injeksjonsmønstre (' OR '1'='1).

        Avansert (regexer og deteksjon)
        
Demoen bruker enkle mønstre for å oppdage alltid-sanne uttrykk i den konstruerte WHERE-klause. Kortversjon:

        
    
          
  • Aritmetikk: mønstre som a + b = c (støtte for +, -, *, /). Demoen evaluerer venstre side numerisk og sammenligner med høyre side; hvis lik, oppdages bypass (f.eks. 1+1=2).
    • 
          
  • Likhet: enkle likheter som 1 = 1 blir også oppdaget.
    • 
          
  • OR-mønster: klassiske injeksjonsmønstre som ' OR '1'='1 blir funnet med en enkel regex.
    • 
        

        
Merk: dette er for demonstrasjon. Virkelige SQL-parsere er mer komplekse, og denne demoen bruker forenklede regler for å gjøre poenget tydelig.
